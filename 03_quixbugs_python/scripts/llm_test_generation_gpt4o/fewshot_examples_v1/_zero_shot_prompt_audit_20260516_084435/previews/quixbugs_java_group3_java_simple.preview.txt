You are generating unit tests for a Java program.

Return ONLY a complete Java JUnit 4 test file.
Do not include Markdown.
Do not include explanations.
Do not include code fences.

Requirements:

QuixBugs Java helper API notes:
- java_programs.Node uses String values. Prefer new Node("A"), new Node("1"), etc.; do not use new Node(1).
- If asserting node.getValue(), compare with a String, e.g. assertEquals("1", node.getValue()).
- Do not invent addSuccessor(...) or addPredecessor(...).
- For graph-style nodes, use node.getSuccessors().add(otherNode) and node.getPredecessors().add(otherNode), or setSuccessors/setPredecessors with ArrayList<Node>.
- If a target method expects ArrayList<Node>, do not pass List<Node>.

- The file must start with: package mypkg;
- Use JUnit 4, not JUnit 5.
- Use imports such as:
  import org.junit.Test;
  import static org.junit.Assert.*;
- The public test class must be named: {TEST_CLASS}
- Test the target class: mypkg.{TARGET_CLASS}
- Prefer deterministic tests.
- Do not use randomness, sleeps, network access, filesystem access, or external services.
- Do not modify the production code.
- Do not use reflection unless strictly necessary.
- Write top-level @Test methods inside the requested test class.

Additional QuixBugs Java constraints:
- Do not invent expected behaviour for ambiguous edge cases.
- Prefer simple, valid, deterministic inputs that are clearly supported by the source.
- Avoid tests that may trigger infinite loops, excessive memory use, sleeps, or very long execution.
- Avoid empty inputs, negative inputs, nulls, or extreme values unless the source clearly supports them.
- Match Java method signatures exactly.
- If a method requires ArrayList<T>, pass new ArrayList<T>(Arrays.asList(...)) or new ArrayList<T>(), not Arrays.asList(...) directly.
- For helper classes such as Node, inspect constructor and setter signatures before using them.
